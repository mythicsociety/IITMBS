Unintended Consequences

When we use a procedure, we expect it to perform a task and maybe return a value. But what if it does more than that? What if it changes the data we gave it in unexpected ways?

These changes to data that exist outside the procedure itself are called side effects. Understanding, managing, and documenting side effects is a critical skill for writing reliable and predictable code.

What are Side Effects?

A side effect occurs when a procedure modifies a parameter or any other data that is outside its own local scope.

Example: The Shuffled Deck

Our `SumMarks` procedure takes a deck of cards as a parameter. To process the cards, it moves them one by one to a "seen" pile. At the end, the original deck is empty! If we restore the cards by moving them back, their original order is reversed. This reordering is a side effect—the procedure returned the correct sum, but it also changed the state of the deck we passed into it.

Good, Bad, and Ugly Side Effects

Not all side effects are bad. We can categorize them:

  • Acceptable/Harmless: Reordering the deck of score cards is a harmless side effect if our next task is just to calculate another sum (since addition doesn't depend on order).
  • Intentional: Sometimes, the side effect is the entire point of the procedure! A procedure called `SortCards` is *supposed* to change the order of the deck. Its main purpose is to produce a side effect.
  • Unacceptable/Harmful: If we had a procedure to find a pronoun's antecedent in a paragraph, reordering the words in the paragraph would be a harmful side effect, as it would break the meaning and logic of subsequent operations.

Interface vs. Implementation

This leads to the crucial concept of a procedure's "contract" or interface. The interface is a promise about what the procedure does.

A Procedure's Interface Specifies:

  • Parameters: What data it accepts as input.
  • Return Value: What data it gives back as output.
  • Side Effects: What changes it is allowed to make to the data it receives.

The actual code inside the procedure is its implementation. The great power of procedures is that we can change the implementation (e.g., to make it faster) as long as we don't break the contract defined by the interface.
